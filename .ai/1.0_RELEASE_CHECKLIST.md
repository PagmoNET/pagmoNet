# PagmoNet 1.0 Release Readiness Checklist

## Context

The repo split is done. pagmo.NET (C#), PagmoNet4j (Java/Kotlin), pagmoNet (shared SWIG/native), and pagmoNet.ipopt now exist as separate repos. The C# side has a mature test suite and production-grade packaging. The Java side has 494 tests passing but several gaps — documentation, JAR native bundling, a wrong license declaration, and less review history — that would make a 1.0 look sloppy or broken to a first user. The split itself introduced significant refactoring that deserves a human review pass.

This checklist tells you what to *do*, in order. It is not "read every file" — it is a targeted workflow: fix blockers, run things, spot-read the risky spots, write the missing docs, wire up infrastructure.

---

## 1. Fix These Blockers First (Before Anything Else)

These are wrong right now and will cause embarrassment or legal issues if shipped:

### 1a. ~~GPL/LGPL mismatch in PagmoNet4j~~ — FIXED
`PagmoNet4j/core/build.gradle.kts` POM license block now correctly declares `LGPL-2.1-or-later`. Fixed in commit `bd8281e`.

### 1b. Do your own license interaction research
Read the actual license texts and understand the interactions before shipping:
- **pagmo2** — LGPL-2.1-or-later (wrapper code can be LGPL, users don't have to open-source their apps)
- **IPOPT** — EPL-2.0 (Eclipse Public License). Read the EPL-2.0 text. Understand: does statically linking EPL code into an LGPL wrapper binary create distribution obligations? Does it require source availability beyond the NOTICE file? Check whether COIN-OR IPOPT explicitly allows GPL/LGPL secondary licensing.
- **NLopt** — LGPL-2.1+. No issue.
- **Boost** — BSL-1.0. No issue (permissive).
- **TBB** — Apache-2.0. No issue (permissive).
- **SWIG-generated code** — the output of a code generator is generally under the user's chosen license; confirm this is your understanding.

Outcome: confirm that LGPL-2.1-or-later is the right label for all four repos given the above. Write a two-sentence explanation in each repo's README of why the license is what it is (especially for pagmoNet.ipopt where EPL bundling is happening).

### 1c. `release.yml` stale paths — CI BLOCKER
`pagmo.NET/.github/workflows/release.yml` was written when `pagmoWrapper/` was local. The native source moved to `pagmoNet/native/` and `build-native.ps1` was updated to detect the submodule — but the workflow's artifact collection paths, cmake invocations, and DLL copy steps still reference `pagmoWrapper/`. Known stale references:
- Line 62: `pagmoWrapper\win-build` in PATH → `pagmoNet\native\win-build`
- Line 126: upload path `pagmoWrapper/build/libPagmoWrapper.so` → `pagmoNet/native/build/libPagmoWrapper.so`
- Lines 164, 184, 191: macOS `pagmoWrapper/build` and `pagmoWrapper/build-x64` → `pagmoNet/native/build` and `pagmoNet/native/build-x64`
- Also: `build-release-artifacts.ps1` is called on Windows (line 54) but that script doesn't exist — needs to be created or the step changed to call `build-native.ps1` directly.

Read the workflow end-to-end and fix every reference. A CI push on a fresh clone is how you'll know it's right.

### 1d. Submodule wiring — CI and fresh-clone BLOCKER
Neither `pagmo.NET` nor `PagmoNet4j` has a `.gitmodules` file. Both CI workflows use `submodules: recursive`. On a fresh GitHub Actions checkout this means: CI will silently succeed to check out without the submodule, then fail cryptically at the cmake step. Fix: `git submodule add <url> pagmoNet` in each language repo (after pushing `pagmoNet` first).

### 1e. Delete duplicate `.ai/`
`pagmo.NET/.ai/` is an identical copy of `pagmoNet/.ai/`. The canonical location is `pagmoNet/.ai/`. Delete the copy from `pagmo.NET` so doc edits don't diverge silently. (The old workspace-level `c:\src\pagmoSharp\.ai\` should also be deleted if it still exists.)

## 2. Run Things and Watch What Happens

Do these in order. Each one surfaces a different class of problem.

### 2a. .NET after the split
The split moved SWIG, native bridge, and Java out of `pagmo.NET`. Run the tests on the current state to confirm the refactoring didn't silently break anything:
```powershell
# In pagmo.NET — build fresh, then test
.\scripts\build-native.ps1 -Configuration Release
dotnet test Tests/Tests.Pagmo.NET/Tests.Pagmo.NET.csproj -l "console;verbosity=normal"
```
Expected: 103 tests green. If any fail, investigate before continuing.

### 2b. .NET examples — golden path
```powershell
dotnet run --project Examples/Examples.Pagmo.NET -- all
```
Watch specifically for: correct champion output from the archipelago scenario, no threading exceptions from the cloning scenario, no `operation not supported` panics. These exercise the paths most likely to have been broken by the split.

### 2c. Java tests on a clean rerun
```powershell
$env:PAGMO4J_NATIVE_DIR = "c:\src\pagmoSharp\PagmoNet4j\pagmoWrapper\win-build"
cd PagmoNet4j
.\gradlew.bat :core:test :kotlin-ext:test --rerun-tasks
```
Expected: 494/494. Any failures after the recent JNI rename are your next priority.

### 2d. Java examples — all six scenarios
```powershell
.\gradlew.bat :examples:run
```
(Or whatever the correct invocation is — check `examples/README.md`.) Run all six: `single`, `archipelago`, `policies`, `cloning`, `kotlin`, `all`. Watch for: the Kotlin scenario outputting sensible champion values, the `cloning` scenario not panicking on multi-threaded evolution, the `policies` scenario completing the migration log.

### 2e. Simulate a fresh-clone build (at least once)
Pick a machine or a temp directory. Clone `pagmo.NET` (or `PagmoNet4j`) *without* your workspace shortcuts, run the build script, run the tests. This is the single most important thing you can do to feel confident: if a stranger clones the repo and follows the README, does it work? This will expose any submodule, path, or toolchain assumption buried in your muscle memory.

## 3. Targeted Code Review

Read these specific files — not the whole codebase, just the spots with the most complexity, the most recent churn, or the least review history.

### 3a. pagmo.NET — post-split refactoring review

| File | What to look for |
|------|-----------------|
| `pagmoExtensions/archipelago.cs` | `WithManagedProblem`, `ExclusiveCloneAdapter`, `_managedProblemCloneRoots`. Confirm roots list is cleared on `Dispose()`. This is the most complex code in the project. |
| `pagmoExtensions/island.cs` | Factory overloads. Confirm no leftover `pagmoWrapper/` path references. |
| `pagmoExtensions/managed_thread_bfe.cs` | Short file — read it entirely. The `Parallel.For` + `ThreadLocal<IProblem>` pattern is the key one. Does it clean up the ThreadLocal on exit? |
| `Pagmo.NET.csproj` — native bundle `<ItemGroup>` blocks | The platform-specific `<Content Include="...">` paths are what gets packed into NuGet. Confirm they still point to the correct build output location after the split. |
| `.github/workflows/release.yml` | Read end-to-end. Every path reference. Does it match the post-split layout? Does the merge step correctly inject Linux/macOS natives into the Windows nupkg? |

### 3b. PagmoNet4j — less review history than .NET

| File | What to look for |
|------|-----------------|
| `core/src/main/java/.../NativeInterop.java` | The `ConcurrentHashMap` root-pinning for problem/algorithm adapters. Confirm that `releaseProblemRoot()` / `releaseAlgorithmRoot()` are called when `delete()` fires and can't be skipped. |
| `core/src/main/java/.../island.java` (extension part) | `constructionRoots` WeakHashMap. Does `close()` / `delete()` release those roots, or is it relying on GC? |
| `core/src/main/java/.../problems/ManagedProblemBase.java` | Look at the virtual method table. What must a user implement? Are the defaults sensible for a beginner? Would a new user know what to override? |
| `kotlin-ext/src/main/kotlin/.../ArchipelagoExtensions.kt` | The `buildArchipelago {}` DSL. Look at what gets closed and when. Anything that creates a native object inside the DSL lambda but returns without closing it is a leak candidate. |
| `examples/src/main/java/Scenarios.java` | Read the `archipelago` and `cloning` scenarios. These are the best documentation of intended use and the first thing a new user will copy. |

### 3c. pagmoNet shared layer

| File | What to look for |
|------|-----------------|
| `swig/shared_core.i` | Skim the `%feature("director")` declarations and the `csdirectorout` / `javadirectorout` exception handler typemaps. These control what happens when a C++ exception crosses into managed code — the most dangerous boundary in the whole project. |
| `native/CMakeLists.txt` | Skim for fragile relative path assumptions now that this file has moved. Does it correctly find pagmo2 and other dependencies via vcpkg? |

## 5. Infrastructure

### Must-do before tagging:

1. **Add pagmoNet submodule** to both `pagmo.NET` and `PagmoNet4j` (after `pagmoNet` is pushed to GitHub).

2. **Java JAR native bundling** — This is the biggest usability gap. Right now a Java user must set `java.library.path` manually; there's no self-contained JAR. Before 1.0, either: (a) add the native DLL to the JAR as a resource under `natives/win-x64/pagmonet4j.dll` and have `NativeLoader` extract it to a temp dir on first use, OR (b) explicitly document in the README that native setup is required and this is a known v1.0 limitation to be fixed in v1.1. Option (b) is the honest choice if (a) would take too long.

3. **Fix release.yml stale paths** (covered in section 1c — repeated here as a reminder it's both a blocker AND an infrastructure item).

4. **Version alignment** — Decide: does C# ship as `1.0.0` or stay `1.0.0-beta.2`? Does Java stay `1.0.0`? Whatever you decide, set both to the same version label before tagging. Consider a `scripts/set-version.ps1 <semver>` that updates both project files at once — cheap insurance against "oops, they're different" after a hotfix.

### Should-do (won't hold up 1.0 but need a plan):

5. **Java Maven Central publish workflow** — PagmoNet4j builds and tests in CI but has no publish job. Maven Central requires GPG signing and OSSRH staging. Decide: Maven Central or GitHub Packages? Add the job before 1.0 tag (even if the actual publish is manual the first time).

6. **Create PagmoNet org on GitHub + Gitea** — Set up the org, transfer/push repos, update remote URLs. This is the "make it public" step and depends on all the above being stable.

7. **Kill the duplicated `pagmoNet` checkout drift** — `pagmoNet` is a nested submodule checked out twice (`Pagmo.Net.ipopt/pagmo.NET/pagmoNet` and `PagmoNet4j.ipopt/PagmoNet4j/pagmoNet`). Every shared native/SWIG change must be mirrored into both working trees by hand, then the gitlinks bumped up two separate parent chains. This drifts in practice: in a single session the base `native/CMakeLists.txt` diverged twice — once adding the `NLopt::nlopt` macOS libm.tbd strip, once refactoring to the generic imported-target sweep — before being re-synced. It is the same species of problem as the duplicate-`.ai/` copy called out in 1e, just harder to notice because it's a build file. Evaluate a structural fix so shared-layer edits land exactly once: `git subtree` for `pagmoNet`, or folding the split back into a monorepo. Post-1.0; needs a design decision, not a quick patch.

## 6. Known Acceptable Gaps — Document, Don't Block On

Write these up in the respective READMEs or ROADMAP so users aren't surprised:

- **`free_form` topology** — dynamic edge add/remove not exposed in Java (pagmo2 API exists but not wrapped)
- **Gradient/hessian callbacks** — not tested end-to-end in Java; low-use path
- **macOS IPOPT** — deferred; NLopt ships instead on macOS
- **Kotlin wrappers for BFE / hypervolume / multi-objective utils** — not yet; use Java API directly
- **Java object lifecycle** — use try-with-resources; if you don't call `close()`, cleanup is finalizer-based and non-deterministic
- **Runtime topology replacement** on managed islands — known limitation; set topology at construction time
- **`namespace pagmo`** in generated C# — the rename to `namespace PagmoNet` is a v1.1 breaking change
