## A. Licensing (highest priority)

Background: the project's license settled after a GPL → LGPL → final arrangement. The final
model is: base packages (`Pagmo.NET`, `PagmoNet4j`) are **MPL-2.0** and statically link
LGPL pagmo2 + NLopt; IPOPT companions (`Pagmo.NET.Ipopt`, `PagmoNet4j.ipopt`) are **EPL-2.0**
pure native payloads; the base never links IPOPT (loads `libipopt` at runtime via
`dlopen`/`LoadLibrary`). The architectural reasoning in `LICENSING.md` (dlopen boundary,
Exhibit-B omission, LGPL headers retained on pagmo-derived files) was reviewed and judged
sound. The two issues below are documentation/packaging gaps, not model problems.

### A1. LGPL relink materials are referenced but do not exist `[CONFIRMED]`
- `pagmo.NET/NOTICE:36-38` — "See the project's LGPL-compliance documentation for how to
  obtain the relinkable components."
- `LICENSING.md:94-95` — "The mechanism for obtaining the relinkable components is described
  in the base packages' LGPL-compliance materials."
- No such document exists anywhere in the repo. The base statically links pagmo2 and NLopt
  (both LGPL), so LGPL-3.0 §4 / LGPL-2.1 §6 require offering a way to relink against modified
  versions. Right now both references point at nothing.
- **Proposed fix (low cost):** the build *is* the corresponding source. Add a short
  `RELINKING.md` (or a section) stating the wrapper builds from public source at the repo,
  and to relink against a modified pagmo/NLopt: edit the vcpkg port under `ports/` and rerun
  `scripts/build-native.ps1`. Repoint the NOTICE + LICENSING.md references at it.
- This was already self-flagged as TODO in `.ai/1.0_RELEASE_CHECKLIST.md` item 2.

- `[OPEN Q]` For static LGPL linking, is "documented from-source rebuild path against public
  source" sufficient, or does the maintainer want to also ship the wrapper object files /
  a linkable intermediate (the more conservative §4(d)(0) "Minimal Corresponding Source +
  object code" reading)? The reviewer believes the documented-rebuild path is adequate given
  everything is open and reproducible, but this is a judgment call the maintainer may have a
  settled opinion on from the earlier licensing work.
  - `[RESOLVED — recommendation]` **Documented rebuild path (a `RELINKING.md`) is sufficient;
    do not ship object files.** LGPL-3.0 §4 offers two routes — §4(d)(0) (ship relinkable object
    files) and §4(d)(1) (a suitable shared-library mechanism). The documented-rebuild route is
    defensible *because* the wrapper source, the vcpkg overlay ports, and `build-native.ps1` are
    all public and reproducible — a recipient can relink against a modified pagmo/NLopt by editing
    the port and rerunning the script. Shipping object-file intermediates adds a per-release
    artifact burden for marginal legal benefit given full source availability. (Judgment call — a
    settled opinion from the earlier licensing work should override this.)

## D. C# → Java drop-in parity

Context for this section: the maintainer's scenario is a user who auto-translates their own
C# (originally written against `Pagmo.NET`) into Java and wants it to work against
`PagmoNet4j` with minimal fuss (namespace/package differences accepted).

Overall finding: **more viable than expected.** The core surface is SWIG-generated in *both*
languages, so pagmo's snake_case names are identical across C# and Java (`fitness`,
`get_bounds`, `get_nobj`, `evolve`, `champion_x`, `get_log_entry`, etc.). Typed log
projections also line up: C# `de.GetTypedLogLines() -> DeLogLine{Generation,...}` maps to
Java `de.getTypedLogLines() -> DeLogLine{generation,...}` under normal casing rules, and the
record members translate cleanly (`ToDisplayString()`→`toDisplayString()`,
`RawFields`→`getRawFields()`, `AlgorithmName`→`getAlgorithmName()`). Verified by comparing
`pagmo.NET/.../Algorithms/de.cs` against the SWIG-injected methods in
`PagmoNet4j/core/src/generated/java/.../de.java`.

The friction points, ordered by likelihood of biting. Recommended deliverable: a short
**"Porting from Pagmo.NET"** doc listing exactly these mappings.

### D1. `Dispose()` → `close()` `[CONFIRMED]`
- C# UDPs override `Dispose()` (`pagmo.NET/.../Problems/ManagedProblemBase.cs:144`); Java
  overrides `close()` (`PagmoNet4j/.../problems/ManagedProblemBase.java:73`,
  `AutoCloseable`).
- A translator that maps `IDisposable→AutoCloseable`/`Dispose→close` is fine; a naïve one
  emits `dispose()` that overrides nothing. **The #1 mapping to document.**

### D2. C# property → Java method `[CONFIRMED]`
- `OptionalSolverAvailability.IsIpoptAvailable` / `.IsNloptAvailable` are C# **properties**
  (`pagmo.NET/.../OptionalSolverAvailability.cs:26,37`) but Java **methods**
  `isIpoptAvailable()` / `isNloptAvailable()`
  (`PagmoNet4j/.../OptionalSolverAvailability.java:22,27`).
- A JavaBean-aware translator (bool `IsX` → `isX()`) lands exactly right; a literal one emits
  `getIsIpoptAvailable()` and breaks. This is the **only** public-surface property, so it's
  contained — but it's precisely where translator behavior diverges.

### D3. Unsigned types → `long` `[CONFIRMED]`
- C# `uint`/`ulong` (`get_nobj()->uint`, `set_seed(uint)`) become Java `long`
  (`get_nobj()->long`, `set_seed(long)`). Unavoidable (no unsigned in Java) and handled by
  any real translator, but overrides must target the `long` signatures.

### D4. IPOPT option-setter alias asymmetry `[CONFIRMED]`
- C# adds convenience `set_integer_option(name, ulong)` forwarding to `set_integer_option_u64`
  (`pagmo.NET/.../Algorithms/ipopt.cs:21`). The Java companion README only documents
  `set_integer_option_u64`.
- Translated C# calling `set_integer_option(...)` won't resolve in Java unless Java grows the
  same alias.
- `[OPEN Q]` Resolve by symmetry: either add the `set_integer_option` convenience alias to the
  Java `ipopt` surface, or drop it from C#. Which direction does the maintainer prefer?
  - `[RESOLVED — neither; the premise is wrong]` **Java already has `set_integer_option(name,
    int)`** — it's generated from the *shared* SWIG `%extend` (`swig/swigInterfaceFiles/algorithms/
    ipopt.i:37`), so both languages get it. Proof it works in Java: the shipping example calls it
    (`Scenarios.java:292`) and so does a Java test (`IpoptSolveWhenAvailableTest.java:48`). The

## F. Full C#↔Java per-algorithm alias diff `[DONE — this closes D4's PARTIAL]`

Ran the complete diff of the handwritten/injected C# algorithm surface
(`pagmo.NET/Pagmo.NET/pagmoExtensions/Algorithms/*.cs`) against the Java surface
(`PagmoNet4j/core/src/generated/java/*.java`, which SWIG builds from the `%typemap(javacode)`
blocks in `swig/Pagmo4jSwigInterface.i`). This supersedes the earlier "risk assessed low"
guess — one finding is **not** low.

### F1. Typed log projections — real gap, 11 of 25 `[HIGH — affects drop-in AND is a feature gap]`
- **C# has `GetTypedLogLines()` for 25 algorithms:** bee_colony, cmaes, compass_search,
  cstrs_self_adaptive, de, de1220, gaco, gwo, ihs, ipopt, maco, mbh, moead, moead_gen, nlopt,
  nsga2, nspso, pso, pso_gen, sade, sea, sga, simulated_annealing, snopt7, xnes. Each also has a
  populated `GetLogLines()`.
- **Java has `getTypedLogLines()` for only 11:** bee_colony, cmaes, compass_search, de, ihs,
  mbh, pso, sade, sea, sga, simulated_annealing.
- **The other 14 Java algorithms** (cstrs_self_adaptive, de1220, gaco, gwo, ipopt, maco, moead,
  moead_gen, nlopt, nsga2, nspso, pso_gen, snopt7, xnes) have **no** `getTypedLogLines()`, and
  their `getLogLines()` is a stub returning `java.util.Collections.emptyList()` (confirmed in
  `xnes.java`, `ipopt.java`). They expose only the raw `get_log_entry(i)` / `get_log_entry_count()`
  primitives. **Note `ipopt` and `nlopt` are in the missing set** — the flagship IPOPT solver's
  `getLogLines()` returns empty in Java while C# projects a typed `IpoptLogLine`.
- **Mechanism / not a regen risk:** the typed-log code is properly in `swig/Pagmo4jSwigInterface.i`
  as per-type `%typemap(javacode) pagmo::<algo>` blocks (de is at ~:213; bee_colony at ~:179), so
  it regenerates cleanly — there's no hand-edited-generated-file wipe risk. The gap is simply that
  only 11 of those javacode blocks were written.
- **Fix (real work, not a doc change):** add a `%typemap(javacode)` block + the `LogLine`
  struct/`%extend get_log_lines()`/vector-template (mirroring `bee_colony.i`) for the other 14,
  then regen + rebuild the JNI. Needs a native build the reviewer can't run locally. **Reported,
  not attempted.** Meanwhile E1 in the Java README was corrected to state this honestly instead of
  advertising parity.

### F2. `grid_search` is C#-only `[MEDIUM — drop-in landmine]`
- C# ships a fully **handwritten** `grid_search` algorithm (`Algorithms/grid_search.cs`,
  `public sealed class grid_search : IAlgorithm`, ctors `grid_search(uint[])` /
  `grid_search(uint)`) — it is not a pagmo UDA and has no SWIG-generated form.
- **Java has no `grid_search` at all.** Translated C# using `grid_search` won't resolve.
- Fix options: port a handwritten `grid_search` to Java, or document it as a C#-only convenience.

### F3. `ipopt.GetLastOptimizationResultCode()` naming `[LOW]`
- C# ipopt exposes `GetLastOptimizationResultCode()` (PascalCase alias) **and** the SWIG
  `get_last_opt_result_code()`. Java has only `get_last_opt_result_code()`. A translator emitting
  `getLastOptimizationResultCode()` misses. Add the alias to Java `ipopt` or document.

### F4. `ipopt.set_integer_option(string, ulong)` overload `[LOW — confirms D4]`
- As D4 concluded: `set_integer_option(string, int)` exists in **both** languages (shared SWIG
  `ipopt.i:37`). The only asymmetry is the C#-only `(string, ulong)` convenience overload
  (`ipopt.cs:21`); int-literal calls bind fine to Java's `(string, int)`. Doc fix only (companion
  README table), no API change. **Done? No** — the companion README table fix was not applied in
  this pass; still worth doing.

### F5. Everything else translates cleanly `[confirmed]`
- The `NotSupportedException`/`UnsupportedOperationException` stubs on ipopt/nlopt/snopt7
  (`set_seed`/`get_seed`/`get_verbosity`) exist in C#; Java's `IAlgorithm` supplies defaults, so
  overrides land. The core SWIG surface (snake_case) and the generic `GetLogLines()`→`getLogLines()`
  contract match across all algorithms (subject to F1's empty-body caveat for the 14).

**Net drop-in verdict, revised:** the core surface and generic log contract port cleanly, but a
translated C# app is **not** guaranteed to compile against Java if it uses (a)
`GetTypedLogLines()` on any of the 14 algorithms, (b) `grid_search`, or (c)
`GetLastOptimizationResultCode()`. The porting guide should list F1–F4 explicitly; F1 is the one
worth closing in code before advertising cross-language parity.
